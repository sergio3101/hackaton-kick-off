# MediConnect — телемедицинская платформа для сети частных клиник

## 1. Контекст

Сеть из 18 частных клиник теряет 25–40% слотов на no-show, очный приём по простым жалобам перегружает регистратуру, а пациенты в малых городах не доходят до профильных специалистов вообще. Колл-центр обзванивает за день не более 60% записанных, расписание ведут в трёх разных системах, скан-копии анализов теряются в почте.

Нужна единая платформа для онлайн-консультаций: пациент находит врача и слот в приложении, проходит видео-приём, получает протокол и направления в личном кабинете. Цель MVP — поднять загрузку врача-консультанта до 80%, удержать no-show ниже 10% и обеспечить пациенту консультацию по острым жалобам в течение 24 часов.

## 2. Целевые пользователи

- **Пациент** — мобильное приложение и web. Записывается, проходит видео-приём, оплачивает, читает протокол, переписывается с врачом. До 50 000 активных в месяц на сеть.
- **Врач-консультант** — web-кабинет. Видит расписание, ведёт приём, заполняет протокол, выписывает направления. ≈300 врачей в системе, 80 одновременно онлайн в пик.
- **Администратор клиники** — управляет расписанием врачей, тарифами, отчётностью.
- **Оператор колл-центра** — записывает пациентов «голосом», помогает с подключением.
- **Партнёр-страховая** — через API проверяет покрытие и подтверждает оплату по ДМС.

## 3. Функциональные требования

### 3.1. Запись на приём

- Поиск слотов по специализации, врачу, языку приёма, цене, ближайшему окну.
- `POST /v1/appointments` с `Idempotency-Key`; защита от двойной записи на тот же слот.
- Отмена пациентом не позднее чем за 2 часа без штрафа; позже — частичный возврат.
- Подтверждение покрытия через API страховой до показа слота как «бесплатного по ДМС».
- Лист ожидания: если врач освобождает слот, очередь автоматически уведомляется по приоритету «острые жалобы → ДМС → очерёдность».

### 3.2. Видео-консультация

- WebRTC через серверный SFU; запись по явному согласию пациента в начале сессии.
- Шеринг экрана пациентом (показать снимок, симптом на коже).
- Фоллбэк-канал — телефонный звонок, если у пациента не подключилось видео в течение 90 секунд.
- Восстановление сессии после кратковременного разрыва (до 30 секунд) без перезаписи участников.
- Качество адаптивно: при потере 5% пакетов — переход на 480p, при 15% — только аудио.

### 3.3. Электронная карта пациента

- Анамнез, аллергии, хронические диагнозы, текущие препараты.
- Протоколы прошлых приёмов (только врач может редактировать; пациент — read-only).
- Вложения: фото, PDF-результаты анализов, DICOM-снимки до 50 МБ.
- Версионирование протокола: правки врача после закрытия приёма доступны только в течение 24 часов и логируются.

### 3.4. Чат и переписка

- E2E-шифрование между пациентом и врачом; ключи генерируются на устройствах.
- Переписка живёт в рамках одной активной заявки (28 дней после приёма), затем переходит в архив.
- Файлы до 25 МБ, превью изображений на клиенте.
- Статусы доставлено/прочитано; индикация набора текста.
- Запрет на отправку пациенту от других пациентов — чат всегда привязан к конкретному приёму.

### 3.5. Уведомления

- Каналы: push (Expo + APNs/FCM), SMS (через двух операторов с round-robin), email.
- Цепочки fallback: запись → push (мгновенно) → SMS через 5 минут, если push не доставлен → email через 30 минут как финальный.
- Учёт `quiet_hours` пациента и его таймзоны: ночные уведомления откладываются до 8:00 локального времени, кроме критичных (отмена приёма врачом).
- Дедупликация по `(notification_id, channel)` — один и тот же ивент не уходит дважды на канал.

### 3.6. Интеграции

- HL7 FHIR R4 для обмена назначениями и результатами с лабораторными партнёрами.
- API страховых (ВСК, Альфа, Согаз) для проверки покрытия и подтверждения оплаты.
- Эквайринг — ЮKassa, рекуррентные платежи для подписок «безлимит на месяц».
- ЕМИАС — выгрузка обезличенной статистики приёмов по запросу регулятора.

## 4. Нефункциональные требования

- 99.95% доступности в часы приёма (07:00–23:00 локального времени клиники).
- p95 ответа основного API — ≤ 250 мс, p99 ≤ 600 мс.
- Видеоканал держит качество без буферизации при потере до 5% пакетов и RTT до 200 мс.
- Поддержка одновременных видео-приёмов — до 200 на сеть в пик.
- ФЗ-152 и ФЗ-323: согласие на обработку ПДн фиксируется явно, медкарта хранится 25 лет, аудит-журнал — 6 лет.
- Шифрование PHI в покое и при передаче, минимум TLS 1.3.
- RPO 5 минут, RTO 30 минут для основной БД.

## 5. Технологический стек

- **Frontend (web)**: Next.js 14 (App Router, React Server Components), TypeScript, TailwindCSS, Zustand. Видео — `livekit-client` поверх SFU.
- **Mobile (пациент)**: React Native + Expo SDK 50, expo-router, expo-notifications.
- **Backend**: Python 3.12, FastAPI, Celery + Redis broker; отдельный SFU-сервис на Go (mediasoup-router, gRPC API).
- **Хранилища**:
  - PostgreSQL 15 — пациенты, врачи, слоты, протоколы; PHI-поля шифруются через pgcrypto.
  - Redis 7 — кэш, очередь Celery, advisory-locks по слотам, сессии WebRTC.
  - S3-совместимое хранилище (MinIO в dev / Yandex Object Storage в prod) — вложения и записи приёмов.
  - ClickHouse — агрегаты для аналитики администратора (в этапе 2).
- **Auth**: Keycloak (SSO для врачей и админов клиник, MFA обязательна), JWT (RS256) для пациентских клиентов; refresh-токены в Redis.
- **Инфраструктура**: Docker Compose в MVP, миграция на Kubernetes в этапе 2. Observability — OpenTelemetry → Grafana Tempo, Loki, Mimir; алертинг — Alertmanager.

## 6. Архитектурные решения и нюансы

- SFU, а не P2P: нужен для записи приёма, для будущих консилиумов (>2 участников) и для надёжного прохождения NAT/корпоративных сетей. P2P оставлен только как fallback при отказе SFU.
- E2E-чат построен на Signal-подобном протоколе (X3DH для рукопожатия + Double Ratchet для сессионных ключей). Сервер хранит шифротекст и метаданные, но не приватные ключи. Конфликт с регуляторным требованием хранить переписку решается key escrow: симметричные ключи сессии шифруются публичным ключом медицинской комиссии и кладутся в HSM; раскрытие — только по запросу регулятора с двойной подписью.
- Idempotent booking — `Idempotency-Key` в заголовке + Postgres advisory-lock на пару `(doctor_id, slot_start)` на время транзакции. Это защищает от overbooking при гонке двух одновременных запросов и от ретраев пациентского клиента при сетевых таймаутах.
- Очередь уведомлений — Celery beat планирует, воркеры идут по цепочке `push → SMS → email` с задержками; respects `patient.quiet_hours` и `patient.timezone`. Каждый шаг проверяет дедуп-ключ `(notification_id, channel)` в Redis с TTL 7 дней, чтобы повторные джобы не дублировали отправку.
- Audit-log живёт в отдельной БД (Postgres) с триггером, запрещающим UPDATE и DELETE; раз в сутки снапшот в S3 с включённым object-lock на 6 лет — это требование ФЗ-323 о неизменяемости медицинской документации.
- Вложения шифруются на клиенте перед загрузкой в S3 (envelope encryption: data-key из KMS, кеш ключей в памяти клиента не дольше длительности сессии). Сервер видит только шифротекст.

## 7. Что в MVP, что отложено

**В MVP**:
- Поиск врача и слота, запись с idempotency, отмена с возвратом.
- Видео-консультация 1:1 с записью и шерингом экрана.
- Электронная карта пациента: анамнез, протоколы, вложения.
- E2E-чат врач↔пациент в рамках активной заявки.
- Цепочки уведомлений push → SMS → email с quiet_hours.
- Оплата картой через ЮKassa, проверка покрытия ДМС.
- Личный кабинет врача с расписанием и протоколами.

**Не в MVP** (этап 2+):
- Консилиумы (>2 участников) и групповые приёмы.
- Полноценный HL7 FHIR-обмен с внешними лабораториями (в MVP — только ручная загрузка PDF).
- Мобильное приложение для врача (в MVP врач работает только в web).
- ML-помощник для расшифровки жалоб и подсказок по дифдиагнозу.
- Интеграция со СМП и государственными медицинскими реестрами.
- Телеметрия с домашних устройств пациента (тонометры, глюкометры, пульсоксиметры).
- Лист ожидания с автонотификацией (в MVP — только ручная переброска оператором).
