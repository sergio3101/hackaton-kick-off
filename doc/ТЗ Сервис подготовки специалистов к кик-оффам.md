# **1\. Контекст и бизнес-боль**

Перед стартом проектов нам нужно готовить специалистов к кик-оффам — проверять знания по стеку клиента и задачам, прогонять сценарии интервью, отрабатывать ответы на типовые вопросы. Сейчас этот процесс ручной: лид тратит время, проверка несистемная, результат субъективен.

Нужен сервис, который сам проводит тренировочное интервью голосом на основе требований проекта, а в конце даёт объективную обратную связь.

# **2\. Цель проекта**

Разработать веб\-сервис, который по описанию проекта или задачи проводит интервью-тренажёр: задаёт вопросы голосом, слушает голосовые ответы, оценивает их по смыслу, поддерживает отдельный режим лайв-кодинга и в конце даёт отчёт по пройденному интервью.

# **3\. Пользователи и сценарии**

## **3.1. Специалист (кандидат)**

•       Регистрируется и входит в личный кабинет.

•       Загружает требования проекта или описание задачи.

•       Выбирает уровень сложности (Junior / Middle / Senior).

•       Проходит голосовое интервью в диалоговом режиме.

•       Переключается в режим лайв-кодинга, когда нужно решить задачу.

•       Получает отчёт по результатам и видит историю своих сессий.

# **4\. Функциональные требования**

## **4.1. Загрузка требований**

•       Пользователь загружает документ с требованиями по проекту или описанием задачи (текст, файл — на усмотрение команды).

•       Система автоматически определяет стек и предметную область из загруженных требований — список тем/технологий для интервью формируется на их основе.

## **4.2. Уровни сложности**

Доступны 3 уровня. От выбранного уровня зависит сложность вопросов и ожидаемая глубина ответа:

•       Junior — базовые понятия, синтаксис, простые сценарии.

•       Middle — применение в реальных задачах, типовые паттерны, типовые подводные камни.

•       Senior — архитектурные решения, trade-offs, сложные сценарии, non-happy path.

## **4.3. Голосовое интервью**

•       Сервис генерирует вопросы на основе загруженных требований и выбранного уровня.

•       Сервис задаёт вопросы голосом.

•       Кандидат отвечает голосом; ответ распознаётся и фиксируется.

•       Формат диалоговый: сервис может задать уточняющий вопрос по ответу.

•       Длительность интервью — 10–15 минут.

•       Должна быть возможность завершить интервью досрочно.

## **4.4. Режим лайв-кодинга**

•       Режим лайв-кодинга переключаемый: во время интервью кандидат/сервис может перейти в него для решения задачи.

•       В интерфейсе доступен редактор кода.

•       Стек языка программирования выбирается по требованиям проекта, загруженным на старте (а не фиксирован заранее).

•       Оценка решения выполняется через LLM-ревью кода (без обязательного запуска и юнит-тестов).

•       После завершения задачи интервью продолжается в голосовом режиме.

## **4.5. Оценка ответов**

•       Ответы оцениваются LLM по смыслу, а не по совпадению ключевых слов.

•       Допустимы градации: верно / частично верно / неверно.

•       Оценка фиксируется для каждого вопроса сессии.

## **4.6. Итоговый отчёт**

После завершения интервью кандидат видит:

•       Список вопросов, заданных в сессии.

•       По каждому вопросу: ответ кандидата (в текстовом виде), оценку системы, короткое описание, почему так оценено.

•       Общую картину: сколько ответов верных, частичных, неверных.

Рекомендации по материалам для подтяжки (ссылки, темы) — не требуются.

## **4.7. Личный кабинет и история**

•       Пользователь авторизуется.

•       История всех пройденных сессий сохраняется в его кабинете.

•       По каждой сессии можно открыть итоговый отчёт повторно.

# **5\. Нефункциональные требования**

•       Решение запускается локально для демо.

•       Интерфейс — веб.

•       Задержка голосового ответа системы должна позволять вести живой диалог (ориентир — пауза не должна восприниматься как «завис»).

•       Инструкция по запуску — обязательна.
