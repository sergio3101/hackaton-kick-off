# FleetOps — диспетчерская платформа городских курьерских служб

## 1. Контекст

Курьерские службы среднего размера (50–300 курьеров на город) сейчас управляют доставками вручную: диспетчер перекидывает заказы в чат, курьеры пишут статусы голосом, маршруты строятся «на глаз». Из-за этого средняя стоимость доставки на 20–30% выше, чем у крупных игроков, а опоздания не отслеживаются.

Нужно построить SaaS-платформу для диспетчеризации: операторы видят карту с курьерами и заказами, система предлагает оптимальное распределение, курьеры используют мобильное приложение с офлайн-режимом, клиент получает ссылку на трекинг. Цель MVP — сократить ручную работу диспетчера в 3 раза и довести on-time доставку до 92%.

## 2. Целевые пользователи

- **Диспетчер** — оператор смены. Видит карту, распределяет заказы, разбирает сбойные ситуации. ≈5 одновременно онлайн на службу.
- **Курьер** — основной пользователь мобильного приложения. Принимает заказ, ведёт по чек-листу, фотографирует подтверждение. До 300 одновременных пользователей на крупный город.
- **Менеджер службы** — отвечает за P&L, смотрит KPI и финансы.
- **Клиент-получатель** — открывает публичную ссылку трекинга, в систему не логинится.
- **Партнёр-магазин** — отдаёт заказы через API (создание/отмена, webhook о статусах).

## 3. Функциональные требования

### 3.1. Приём заказов

- REST API для партнёрских магазинов: `POST /v1/orders`, `DELETE /v1/orders/{id}`, ключ авторизации в заголовке.
- Webhook обратно в магазин при смене статуса (`accepted`, `picked_up`, `delivered`, `failed`).
- Ручное создание заказа диспетчером.
- Импорт CSV из легаси-системы (один раз в час, по cron).

### 3.2. Жизненный цикл заказа

Состояния: `created` → `assigned` → `picked_up` → `in_delivery` → `delivered` либо `failed`/`returned`. Каждое изменение пишется в журнал с указанием актора (курьер/диспетчер/система) и геопозицией.

Атрибуты заказа:
- адрес забора и доставки (геокодирование при приёме);
- временное окно доставки;
- габариты и вес;
- приоритет (`standard` / `express`);
- цена доставки и комиссия курьера.

### 3.3. Распределение и маршрутизация

- Автораспределение учитывает: текущее местоположение курьера, его загрузку, временное окно, габариты, рейтинг.
- Алгоритм первой версии — жадный с пересчётом каждые 30 секунд; модель улучшения — VRPTW через внешний сервис в этапе 2.
- Диспетчер может переопределять решение системы (drag-and-drop на карте).
- Группировка соседних заказов в один «батч» курьеру.

### 3.4. Мобильное приложение курьера

- Offline-first: заказы и треки хранятся локально, синхронизируются при появлении сети.
- Чек-лист по статусам: «выехал» → «забрал» → «привёз» → «закрыл». Для подтверждения вручения — фото и подпись пальцем.
- Фоновая отправка GPS-точек (раз в 15 секунд при активной доставке, раз в 60 — в простое).
- Push-уведомления о новом заказе, изменении маршрута, отмене.
- Поддержка iOS 15+ и Android 9+.

### 3.5. Карта диспетчера

- Real-time позиции курьеров и заказов на карте города (OpenStreetMap-тайлы внутри VPN, без публичных API).
- Heatmap нагрузки по районам.
- Цветовая индикация SLA: зелёный/жёлтый/красный.
- Фильтры по статусу, курьеру, партнёру, приоритету.

### 3.6. Трекинг для клиента

- Публичная ссылка `/t/{public_id}` без авторизации, токен ограничен по времени.
- Показ примерного ETA, статуса заказа и обезличенного имени курьера.
- Возможность связаться с курьером через masked-номер (через интеграцию с виртуальной АТС).

### 3.7. Финансы и отчётность

- Дашборд менеджера: заказы в день, on-time %, средний чек, утилизация курьеров.
- Расчёт комиссии курьера по итогам смены (пер-заказ + бонус за on-time).
- Выгрузка реестров для бухгалтерии в формате 1С (XML).

## 4. Нефункциональные требования

- 95-й перцентиль времени ответа основного API — ≤ 200 мс.
- Поток GPS-точек — до 10 000 событий/сек на пиковом городе. Хранить детально 30 дней, в агрегатах — 2 года.
- Доступность — 99.9% в рабочие часы службы (06:00–24:00 локального времени).
- RPO — 5 минут, RTO — 30 минут.
- Локализация — русский, казахский, узбекский (текстовые ресурсы в i18n-файлах, динамическое переключение в UI).
- PII (телефоны клиентов и курьеров) — шифрование в БД, маскирование в логах.

## 5. Технологический стек

- **Backend**: Go 1.22, фреймворк Echo, gRPC между внутренними сервисами. Сервисы: `orders-api`, `dispatch-engine`, `geo-stream`, `notifier`, `billing`.
- **Frontend (диспетчер)**: Vue 3 + TypeScript, Vite, Pinia, TailwindCSS, MapLibre GL для карты.
- **Mobile (курьер)**: Flutter 3.x, Dart, локальная БД — Drift (SQLite). Push — FCM (Android), APNs (iOS).
- **Хранилища**:
  - PostgreSQL 15 — основная БД (заказы, пользователи, журналы).
  - ClickHouse — аналитика и GPS-треки в долгом хранении.
  - Redis 7 — кэш и распределённые блокировки на распределении заказов.
  - Kafka 3.x — шина событий между сервисами и стрим GPS-точек.
- **Инфраструктура**: Kubernetes (Yandex Managed K8s), Helm, ArgoCD. Tracing — Jaeger, метрики — Victoria Metrics, логи — Loki.
- **Auth**: собственный сервис на JWT (RS256), refresh-токены в Redis. SSO для крупных партнёров — OIDC.

## 6. Архитектурные решения и нюансы

- Сервис `dispatch-engine` — единый владелец стейта распределения; остальные сервисы общаются с ним только через API. Это нужно, чтобы не словить race condition при параллельных назначениях одного курьера.
- Поток GPS-точек идёт `mobile → geo-stream (gRPC) → Kafka → ClickHouse`. В Postgres попадает только последняя позиция курьера (через materialized state в Redis с записью в Postgres раз в 5 секунд) — иначе Postgres задохнётся под write-нагрузкой.
- Идемпотентность критичных API: на `POST /orders` партнёр обязан передавать `Idempotency-Key`; ключи живут 24 часа в Redis.
- Distributed lock для назначения заказа курьеру — Redlock на Redis. TTL 30 секунд, автопродление, если процесс ещё работает.
- В мобильном приложении синхронизация — vector clock на уровне записей: каждая локальная мутация имеет `(device_id, lamport_ts)`, сервер делает merge по правилу last-write-wins на поле + аудитный журнал.
- Все internal-эндпоинты закрыты mTLS внутри K8s; снаружи доступен только API-gateway.

## 7. Что в MVP, что отложено

**В MVP**:
- Приём заказов из API и вручную, импорт CSV.
- Полный жизненный цикл заказа, журнал и аудит.
- Жадный алгоритм распределения, ручное переопределение диспетчером.
- Мобильное приложение курьера с офлайн-режимом и фото-подтверждением.
- Карта диспетчера с реалтаймом, базовые фильтры.
- Публичная страница трекинга для клиента.
- Дашборд менеджера, выгрузка реестров.
- Локализация: русский.

**Не в MVP** (этап 2+):
- VRPTW-маршрутизация через внешний оптимизатор.
- Masked-звонки клиент↔курьер (нужен договор с виртуальной АТС).
- Heatmap-аналитика и предсказание спроса.
- Локализации kk/uz.
- ML-скоринг курьеров и динамические комиссии.
- SSO через OIDC для крупных партнёров.
