# Customer Support Hub — внутренний портал обращений клиентов

## 1. Контекст

В компании сейчас обращения клиентов разбросаны по почте, мессенджерам и телефонным звонкам. Поддержка теряет тикеты, СЛА (SLA) не отслеживается, аналитики по причинам обращений нет.

Нужно построить внутренний веб-портал, в который операторы поддержки фиксируют обращения, ведут переписку с клиентом, передают тикеты между отделами и видят аналитику. Цель — сократить среднее время реакции с 4 ч до 30 мин и обеспечить прозрачность по SLA для руководителя.

## 2. Целевые пользователи

- **Оператор поддержки** (≈40 человек) — основной пользователь. Ведёт тикеты, отвечает клиенту.
- **Тимлид** (≈6) — назначает тикеты, видит загрузку команды, контролирует SLA.
- **Аналитик** (2) — строит отчёты по типам обращений, конверсиям, нагрузке.
- **Клиент (внешний пользователь)** — взаимодействует с системой только через email и виджет на сайте.

## 3. Функциональные требования

### 3.1. Канал поступления обращений

- Импорт писем с почтового ящика `support@company.ru` (IMAP, опрос раз в минуту). Каждое письмо превращается в новый тикет либо привязывается к существующему по `In-Reply-To`.
- Виджет на сайте компании — отдельный JS-скрипт, открывающий чат. Сообщения уходят в портал по REST API.
- Ручное создание тикета оператором (например, после звонка).

### 3.2. Жизненный цикл тикета

Состояния: `new` → `assigned` → `in_progress` → `waiting_customer` → `resolved` → `closed`. Переходы фиксируются в журнале, видны в карточке тикета.

Каждому тикету присваивается:
- категория (выбирает оператор или предсказывает ML-классификатор; для MVP — ручная);
- приоритет (`low`/`normal`/`high`/`urgent`);
- ответственный оператор;
- SLA по приоритету (high — 1 ч, urgent — 15 мин и т. д.).

При изменении статуса автоматически отправляются уведомления:
- клиенту по email при `resolved` — с просьбой подтвердить решение;
- ответственному оператору в Slack при назначении.

### 3.3. Карточка тикета и переписка

- Хронология сообщений: входящие, исходящие, внутренние комментарии (видны только команде).
- Прикрепление файлов — до 25 МБ, типы `pdf|png|jpg|docx|zip|txt|log`.
- Шаблоны ответов — у каждого оператора свой набор; шаблоны поддерживают подстановку (имя клиента, номер заказа).
- Slash-команды в редакторе: `/назначить`, `/закрыть`, `/приоритет`, `/шаблон`.

### 3.4. Очереди и распределение

- Главная страница оператора — его «очередь»: тикеты, назначенные на него, отсортированные по сроку SLA.
- «Общая очередь» с фильтрами: категория, приоритет, ожидающие назначения.
- Тимлид может переназначать тикеты, делать массовые действия (закрыть выбранные, сменить категорию).
- Автораспределение «по кругу» в рамках команды (можно отключить).

### 3.5. Аналитика

- Дашборд: количество тикетов в день/неделю/месяц, разбивка по категориям, доля нарушений SLA, среднее время первого ответа.
- Отчёт по операторам: количество, среднее время решения, удовлетворённость клиента (CSAT по пост-опросу).
- Экспорт в CSV.

### 3.6. Авторизация

- SSO через корпоративный Keycloak (OIDC). Внутренних паролей нет.
- Роли: `agent`, `lead`, `analyst`, `admin`. Конфигурируются на стороне Keycloak, читаются из claim'а `roles`.

## 4. Нефункциональные требования

- Среднее время отклика API — не более 300 мс на 95-м перцентиле.
- В пиковые часы — 200 одновременных пользователей, 30 RPS на чтение.
- Хранение переписки — 5 лет. Файлы — в S3-совместимом хранилище (внутренний MinIO).
- Аудит — каждое изменение тикета пишется в неизменяемый журнал, доступный аналитику.
- Доступность — 99.5% в рабочие часы (08:00–22:00 МСК).
- Логи и метрики — Prometheus + Grafana, трейсинг — OpenTelemetry в Tempo.

## 5. Технологический стек

- **Frontend**: React 18 + TypeScript, Vite, TanStack Router, TanStack Query, Tailwind CSS, ProseMirror для редактора сообщений.
- **Backend**: Python 3.12, FastAPI, SQLAlchemy 2 (async), Alembic, Pydantic v2.
- **Хранилища**:
  - PostgreSQL 16 — основная БД (тикеты, пользователи, журнал, шаблоны).
  - Redis 7 — кэш + очередь задач (RQ).
  - MinIO (S3 API) — файлы вложений.
- **Интеграции**:
  - IMAP-клиент для импорта писем (`aioimaplib`).
  - Slack — webhook + slash-commands.
  - Keycloak — OIDC.
- **Инфраструктура**: Docker Compose в dev, Kubernetes (внутренний кластер) в prod. CI/CD — GitLab.

## 6. Архитектурные решения и нюансы

- Импорт писем выполняется отдельным воркером (`worker-mail`), а не основным приложением, чтобы тяжёлые письма с большими аттачами не блокировали API.
- Отправка уведомлений (email клиенту, Slack оператору) — через RQ-очередь, чтобы при падении внешних сервисов основное приложение продолжало работать.
- Поиск по тикетам — встроенный в Postgres (`tsvector` + `GIN` индексы по полям `subject`, `body_concat`); ElasticSearch не используем.
- Для real-time обновления карточки тикета (когда коллега пишет внутренний комментарий) — Server-Sent Events на канале `/api/tickets/{id}/events`.
- Все таймстемпы в БД — `timestamptz` в UTC. Преобразование в часовой пояс пользователя — на фронте.

## 7. Что в MVP, что отложено

**В MVP**:
- Импорт почты, ручное создание тикетов.
- Жизненный цикл, карточка, переписка, файлы.
- Очереди, переназначение, фильтры.
- SSO Keycloak.
- Базовый дашборд (тикеты в день, SLA нарушения, по категориям).
- Email-уведомления клиенту.

**Не в MVP** (этап 2+):
- Виджет на сайте.
- Slack-интеграция.
- ML-классификатор категорий.
- CSAT-опросы.
- Шаблоны со сложной подстановкой и условной логикой.
